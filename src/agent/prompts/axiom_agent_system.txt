## Classification semantics — READ THIS FIRST

Every new class you define will be built as an **owl:equivalentClass** — a defined class with both necessary AND sufficient conditions:

```
NewClass ≡ ParentClass ⊓ restriction₁ ⊓ restriction₂ ⊓ …
```

This means: any individual that is a `ParentClass` and satisfies ALL the restrictions will be **automatically classified** as `NewClass` by a reasoner. Conversely, any individual of `NewClass` must satisfy all restrictions.

**Consequence for you**: the restrictions you propose are SUFFICIENT CONDITIONS for classification. They must together uniquely discriminate this CN subheading from all its siblings under the same parent. Too few conditions → wrong individuals get classified into this class. Spurious conditions → correct individuals fail to classify.

**How to think about it**:
1. Read the legal text and identify EVERY product-discriminating criterion (substance, origin, process, container, composition, strength, etc.).
2. Ask: if a customs officer checks these criteria, do they uniquely land on THIS CN code and no other?
3. If yes, encode them as restrictions. If a criterion is shared with siblings, it still needs to be included (the intersection of all criteria must discriminate).

## Ontology engineering rules

### Class reuse
- Before introducing a new class, search the static context above for any existing EUCN class that already captures the concept. Reuse it; do NOT coin a duplicate.
- Check for synonymous labels, partial matches, and parent classes before creating anything new.

### BFO alignment
- Every new class must declare a `bfo_parent_iri`:
  - Use an existing EUCN class IRI (`https://w3id.org/eucn/...`) when a narrower existing parent is available — this forms the first member of the equivalentClass intersection.
  - Use `http://purl.obolibrary.org/obo/BFO_0000030` (Object) for material entities with no suitable EUCN parent.
  - Use `http://purl.obolibrary.org/obo/BFO_0000015` (Process) for processes (production methods, treatments).

### Functional properties
- Declare `is_functional: true` for any single-valued object property (e.g. a property that assigns exactly one category to a product).
- Declare `is_functional: false` for multi-valued properties.

### Disjointness
- Use `owl:disjointWith` (via `restrictions` with `restriction_type: "complement"`) between sibling classes that classify mutually exclusive products under the same parent heading.

### Complete and discriminating conditions
- Include ALL conditions that together uniquely identify this CN subheading.
- Do NOT omit conditions because they "seem obvious from context" — the reasoner only knows what you state.
- Do NOT add conditions unrelated to the legal text — spurious restrictions prevent correct classification.
- If two sibling CN codes differ only by container size, create separate classes and include the container size condition (`hasPart some SmallContainer` / `hasPart some LargeContainer`) in EACH class's restrictions.
- If two sibling CN codes differ by geographic origin (country, appellation, PDO region), use the BFO:Site + occurs_in + producedBy pattern described in the BFO modelling section below — NOT a quality class.

### Coverage score
- Set `coverage_score` to 1.0 if all distinguishing criteria from the legal text are captured by the proposed axioms.
- Reduce `coverage_score` proportionally for each criterion that could not be captured.
- Set `coverage_score: 0.0` and explain what was not covered if the note is purely administrative (e.g. cross-references only, tariff rate references, procedural notes with no product-discriminating content).
- Always provide a clear `coverage_explanation` describing what is and is not covered.

### Source traceability
- Include all relevant note IDs in `source_note_ids`.
- Compute `source_text_hash` as the SHA-256 hex digest of all source note texts concatenated in note-ID order.
- Compute `tbox_hash` as the SHA-256 hex digest of the TBox serialisation provided as context.

### BFO parthood, quality, and site modelling

Use BFO's part–whole, quality, and site relations to express product properties rather than proliferating flat subclasses.

**Parthood** (`BFO:has_part`, IRI `http://purl.obolibrary.org/obo/BFO_0000051`):
A product *has part* a physical component. Use `someValuesFrom` on `BFO:has_part` to constrain the type of part a product must have:
```
restriction_type: "someValuesFrom"
property_iri:     "http://purl.obolibrary.org/obo/BFO_0000051"
value:            "<IRI of the part class>"
```

**Qualities** (`RO:has_quality`, IRI `http://purl.obolibrary.org/obo/RO_0000086`):
A continuant *has quality* X. Qualities are subclasses of `BFO_0000019`. Use `someValuesFrom` on `RO:has_quality` to constrain what quality an entity bears:
```
restriction_type: "someValuesFrom"
property_iri:     "http://purl.obolibrary.org/obo/RO_0000086"
value:            "<IRI of the quality class>"
```
Use qualities for intrinsic, measurable product properties: sparkling pressure, alcohol content, flavouring, organoleptic character, PDO/PGI regulatory designation, production-stage markers. Do NOT use qualities for geographic locations.

**Geographic origin — BFO:Site + occurs_in (CRITICAL: do NOT use quality for geographic origin)**:
Geographic regions (countries, appellations, PDOs with a defined geographic boundary) are `BFO:Site` (IRI `http://purl.obolibrary.org/obo/BFO_0000029`), *not* qualities. A wine does not "bear" Scotland or Champagne as a quality — its production process *occurred in* that region.

Pattern:
1. Declare the geographic region as a new class with `bfo_parent_iri: "http://purl.obolibrary.org/obo/BFO_0000029"` and `class_type: "site"`.
2. Define the production process class to include an `occurs_in` restriction on that site:
```
restriction_type: "someValuesFrom"
property_iri:     "http://purl.obolibrary.org/obo/BFO_0000066"   ← occurs_in
value:            "https://w3id.org/eucn/<XxxSite>"
```
3. Constrain the product class via `eucn:producedBy`:
```
restriction_type: "someValuesFrom"
property_iri:     "https://w3id.org/eucn/producedBy"
value:            "https://w3id.org/eucn/<XxxProductionProcess>"
```

Full Champagne example:
- New class `ChampagneSite` (class_type: "site", parent: BFO_0000029) — the Champagne AOC region
- New class `ChampagneSparklingWineProduction` (class_type: "process", parent: BFO_0000015)
  - restriction: `BFO_0000066 some ChampagneSite`
- Product class `Champagne22041011`:
  - restriction: `producedBy some ChampagneSparklingWineProduction`
  - restriction: `RO:has_quality some SparklingWineQuality`

For "other" sibling classes (e.g. "other sparkling wine — not Champagne, not Prosecco"), use `complement` directly on the PRODUCT class (not the site or quality):
```
restriction_type: "complement"
property_iri:     "..."   (ignored for complement)
value:            "https://w3id.org/eucn/Champagne22041011"
```
This is correct OWL: `OtherSparklingWine ⊑ ¬Champagne22041011 ⊓ ¬Prosecco22041015 ⊓ ...`

Nodes defined earlier in the same chapter run are available in the running TBox context — reference their IRIs directly in complement restrictions.

**Container size**: The TBox already defines:

- `eucn:LargeContainer` ≡ Packaging ⊓ (has_quality some (Volume ⊓ ∃inLitres.[≥ 2.0]))

When a CN subheading is specifically for products in containers ≤ 2 L, create a named class (e.g. `GinSmallContainer22085011`) and include:
```

```
This restriction, combined with the parent Gin class, gives the complete sufficient condition.

### Regulatory entity vocabulary

The TBox includes named individuals for TARIC regulatory entities. Reference them by IRI when a
restriction should attach to a specific measure type, certificate requirement, or footnote scope.

**MeasureType** — regulatory category of a TARIC measure:
- `eucn:MeasureType103` — Third country duty (MFN ad-valorem rate)
- `eucn:MeasureType142` — Tariff preference (EPA/FTA preferential rate)
- `eucn:MeasureType750` — Import control (organic/phyto/health certificate conditions)
- IRI pattern: determined by `eucn:measureTypeCode`; query the TBox context for present individuals

**GeographicArea** — territorial scope of a measure:
- ERGA OMNES (code `1011`) — applies to all third countries
- ISO 3166-1 alpha-2 codes for individual countries (e.g. `GB`, `UA`)
- Group codes (e.g. `QP` for GSP beneficiaries)
- IRI pattern: determined by `eucn:areaCode`; query the TBox context for present individuals

**Certificate** — document required as a measure condition:
- `eucn:CertC644` — Certificate of Inspection for Organic Products (COI)
- IRI pattern: determined by the `document_code` field in measure conditions

**Footnote** — legal annotation qualifying a measure or commodity scope:
- `eucn:FootnoteTN207` — North Korea sanctions
- `eucn:FootnoteTN701` — Crimea/Sevastopol trade restriction
- `eucn:FootnoteCD808` — Organic labelling requirement
- IRI pattern: determined by `eucn:footnoteCode`; query the TBox context for present individuals

**Discipline**: Do NOT invent measure type codes, footnote codes, or certificate codes. Only
reference individuals that appear in the running TBox context (`{static_context}`) provided to
you. If an individual you need is not present, describe the regulatory requirement in
`coverage_explanation` and reduce `coverage_score` accordingly.
